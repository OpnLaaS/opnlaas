## **\[20\] Sprint review**

### Project and sprint requirements

Implemented and deployed by end of project:

* Two features, ~3-4 stories per feature

Implemented by end of this sprint:

* at least 2 completed stories per group
* at least one completed task/issue per team member

### Sprint review

Each team will conduct a sprint review in class.  They will document their meeting.  After class, they will create demo videos.

Rubric:

* [8] **Meeting and document**:  
  * \[2\] sprint goal and responsibilities (use 2x2 matrix)  
  * \[4\] review sprint backlog (by feature, then by story)   
    * Plan out demo video
  * \[2\] next steps  

* \[8\] **Demo video**  
  - [4] progress shown - (new features only); see requirements above.
  - [2] contributions - everyone shows their own work
  - [2] mechanics - preparation, organization, etc.

* \[4\] **Git/Gitlab**  
  - [2] issue board - completed issues are moved to closed.
  - [2] repo - repo is well-organized and has a helpful README.md
    - contributions - everyone has pushed code for their work items.

Submission:

* Submit review document to Canvas.

## Backlog review

Review the stories and work items selected for the sprint.  

Completed stories have all work tasks done and tested, e.g.:
* backend work / unit tests
* frontend work / user tests
* integration work / integration tests.

You're not required to have anything deployed yet.

## Demo

Key points:

* Audience is your grader - someone who is familiar with the project
* Demo by story, then by developer (backend/frontend). Each individual should present their own work.
* Demo only the two stories you've finished - don't show work from partially completed stories unless you haven't finished two.

Mechanics:

* The demo should be very concise (~5 min).  
* Create a Teams meeting, record, share your screen.  Make sure that you plan before you record.  

### Story X

* Intro:

  - Describe the story that is being demoed / tested.
  - (You can show backend or frontend first - up to you)

* Backend demo (internal or external API)
  - **Introduce yourself**
  - show testing script(s) and explain what is being tested
    - e.g. unittests that are executing your API functions (internal) OR calling your API endpoints (external)
  - run test(s) and show results
  
* frontend demo:
  - **Introduce yourself**
  - show testing script(s) and explain what is being tested
    - e.g. written instructions for a user to complete one or more stories, or (optionally) automated UI tests
  - demonstrate the story by follow the instructions
